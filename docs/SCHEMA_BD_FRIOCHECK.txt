1. TABLA: tenants (Multi-tenant)
================================================================================
Descripción: Registro de clientes/tenants del sistema

Columnas:
  - id (UUID PRIMARY KEY DEFAULT gen_random_uuid()) - Identificador único
  - name (VARCHAR NOT NULL) - Nombre del tenant
  - slug (VARCHAR UNIQUE NOT NULL) - URL-friendly identifier
  - description (TEXT NULL) - Descripción
  - logo_url (VARCHAR NULL) - URL del logo del tenant
  - is_active (BOOLEAN DEFAULT true) - Estado activo/inactivo
  - created_at (TIMESTAMP DEFAULT CURRENT_TIMESTAMP) - Fecha creación
  - updated_at (TIMESTAMP DEFAULT CURRENT_TIMESTAMP) - Última actualización
  - deleted_at (TIMESTAMP NULL) - Soft delete

Índices: slug (UNIQUE)

Relaciones:
  ← users (1:N)
  ← machines (1:N)
  ← nfc_tags (1:N)
  ← visits (1:N)
  ← work_orders (1:N)
  ← tickets (1:N)
  ← attachments (1:N)
  ← sync_queue (1:N)

2. TABLA: users (Usuarios del Sistema)
================================================================================
Descripción: Usuarios con roles y permisos específicos

Columnas:
  - id (UUID PRIMARY KEY DEFAULT uuid_generate_v4()) - Identificador único
  - tenant_id (UUID NOT NULL) → FK: tenants.id [CASCADE] - Tenant del usuario
  - email (VARCHAR UNIQUE NOT NULL) - Email único
  - password_hash (VARCHAR NOT NULL) - Hash bcrypt
  - first_name (VARCHAR NOT NULL) - Nombre
  - last_name (VARCHAR NOT NULL) - Apellido
  - phone (VARCHAR NULL) - Teléfono
  - role (users_role_enum NOT NULL) - Rol del usuario
    ├─ ADMIN: Administrador del tenant
    ├─ SUPPORT: Soporte técnico
    ├─ VENDOR: Vendedor/Proveedor
    ├─ RETAILER: Minorista/Cliente
    ├─ TECHNICIAN: Técnico de servicio
    └─ DRIVER: Conductor/Transportista
  - fcm_tokens (TEXT NULL) - Tokens FCM para notificaciones push
  - active (BOOLEAN DEFAULT true) - Estado activo
  - created_at (TIMESTAMP DEFAULT CURRENT_TIMESTAMP)
  - updated_at (TIMESTAMP DEFAULT CURRENT_TIMESTAMP)
  - deleted_at (TIMESTAMP NULL)

Índices:
  - email (UNIQUE)
  - tenant_id

Foreign Keys:
  - tenant_id → tenants(id) ON DELETE CASCADE

Relaciones:
  → tenants (N:1)
  ← machines (1:N) - assigned_user_id
  ← visits (1:N) - user_id
  ← work_orders (1:N) - assigned_user_id
  ← tickets (1:N) - reported_by_id, resolved_by_id
  ← attachments (1:N) - uploaded_by_id
  ← sync_queue (1:N) - user_id

================================================================================
RESTRICCIONES DE INTEGRIDAD
================================================================================

✓ KEY CONSTRAINTS:
  - Todas las PKs son UUID auto-generados
  - Todas las FKs tienen tenant_id para multi-tenancy
  - Soft deletes con deleted_at en todas las tablas

✓ CASCADE & RESTRICT RULES:
  tenant_id → tenants [CASCADE]: Eliminar tenant elimina todos sus datos
  assigned_user_id → users [RESTRICT]: No eliminar usuario si está asignado
  user_id → users [RESTRICT]: No eliminar usuario con visitas activas
  machine_id → machines [RESTRICT]: No eliminar máquina con análisis activo

✓ MULTI-TENANCY:
  Cada tabla tiene tenant_id como FK a tenants.id
  Todas las queries deben filtrar por tenant_id
  Índices incluyen tenant_id para performance

✓ SOFT DELETE:
  deleted_at IS NULL en todas las queries READ
  Para recuperar datos: actualizar deleted_at a NULL
  Para hard delete: eliminar físicamente si necesario

✓ TIMESTAMPS:
  created_at: Al crear registro (DEFAULT CURRENT_TIMESTAMP)
  updated_at: Al crear y actualizar (DEFAULT CURRENT_TIMESTAMP)
  deleted_at: NULL = Activo, NOT NULL = Eliminado

NOTAS IMPORTANTES
================================================================================

1. MULTI-TENANCY:
   - Filtrar SIEMPRE por tenant_id en queries
   - Usar TenantGuard para validar tenant en request
   - CASCADE delete para tenant elimina datos en cascada

2. SOFT DELETE:
   - NO eliminar registros físicamente
   - Usar deleted_at IS NULL para filtrar activos
   - Restaurar: update deleted_at a NULL

3. GPS COORDINATES:
   - NUMERIC(10,8) para latitud (-90 a 90)
   - NUMERIC(11,8) para longitud (-180 a 180)
   - Precisión de 8 decimales ≈ 1.1 mm

4. NFC SECURITY:
   - Validar UID format (hexadecimal)
   - Verificar checksum de integridad
   - Evitar reusos de UIDs
   - Bloquear etiquetas después de uso

5. FILE STORAGE:
   - URLs almacenadas en azure_blob_url
   - Máximo size en file_size_bytes
   - Type validation: FOTO, DOCUMENTO, FIRMA, VIDEO
   - Category para clasificación

6. PERFORMANCE:
   - Índices en tenant_id, status, created_at
   - Considerar particionamiento por tenant_id si crece
