# FrioCheck - Esquema de Base de Datos

## Descripción General

Sistema de gestión integral para cámaras de frío y equipos de refrigeración. Controla visitas técnicas, mantenimiento, ventas, mermas, inventario y sincronización offline.

**Tecnologías:** PostgreSQL 17 (Azure), TypeORM, NestJS

---

## Tablas

### 1. TENANTS (Empresas/Clientes)

Empresas o clientes que utilizan el sistema. Cada tenant tiene sus propios usuarios, máquinas, tiendas, etc.

| Campo | Tipo | Restricciones | Descripción |
|-------|------|-------------|------------|
| id | UUID | PK | Auto-generado | Identificador único |
| name | VARCHAR(255) | NOT NULL | Nombre de la empresa |
| slug | VARCHAR(255) | UNIQUE, NOT NULL | Identificador en URL (ej: "superfrio") |
| description | TEXT | NULL | Descripción de la empresa |
| logo_url | VARCHAR(500) | NULL | URL del logo |
| is_active | BOOLEAN | DEFAULT TRUE | Si está activa |
| created_at | TIMESTAMP | DEFAULT NOW | Fecha de creación |
| updated_at | TIMESTAMP | DEFAULT NOW | Última modificación |
| deleted_at | TIMESTAMP | NULL | Soft delete |

**Relaciones:**
- 1:N → users (cada usuario pertenece a un tenant)

## Notas de Implementación

### Multi-tenant
- Todas las tablas tienen `tenant_id`
- Queries siempre incluyen `tenant_id` como filtro
- Middleware de NestJS valida acceso

### Soft Delete
- Todas las tablas tienen `deleted_at`
- Queries filtran `deleted_at IS NULL`

### GPS
- Máxima distancia: 100 metros entre NFC y machine
- Usar Haversine formula para calcular distancia

### Offline
- App móvil guarda local y sincroniza después
- Sync queue procesa en orden FIFO
- Retry automático con backoff exponencial
